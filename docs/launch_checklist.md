# SpecPilot AI 출시 체크리스트

이 문서는 제품 기획서 기준으로 현재 구현 상태와 실제 공개 전 확인할 항목을 나눈다.

## 현재 공개 가능한 데모 범위

- 웹 UI에서 데스크톱 PC와 노트북 분석 실행
- 분석 전 조건 진단으로 누락 조건, 추가 질문, 추천 필수/제외 조건 확인
- 후보 5개 비교
- TOP 3 추천과 제외 후보 2개 표시
- 실구매가 계산: 상품가, 배송비, 조립비, 쿠폰, 카드 할인
- 호환성 체크: 소켓, 파워, GPU 장착 공간, 쿨러 높이, RAM, 노트북 GPU/무게/배터리
- 리뷰 리스크와 벤치마크 근거 표시
- 후보별 가격, 리뷰, 벤치마크, 호환성, 출처 신뢰 근거 팩 표시
- 후보별 옵션/사양 검수표로 판매 페이지 옵션명과 핵심 사양 대조
- 공개 공유 전용 검토 브리프와 복사용 검토 문구 표시
- 구매 판정, 확신도, 위험 플래그, 다음 행동 표시
- 예산 절감, 성능 우선, 안전 우선 대안 시나리오 비교
- 필수 조건과 제외 조건의 후보별 충족 매트릭스 표시
- 예산 10% 절감, 예산 10% 여유, 조건 강화 스트레스 테스트 표시
- 후보별 현재가, 목표가, 적정가 밴드, 변동 리스크, 결제 트리거 표시
- 결제 전 실행 단계, 판매자 확인 질문, 공유 검토 문구 표시
- 가격 알림 목표가 미리보기
- 저장 리포트 생성과 조회
- 저장 리포트 공개 공유 링크 생성과 공개 리포트 페이지
- 완료 리포트 템플릿, 수신자 그룹, unsubscribe 제외, 발송 전 미리보기, 추적 픽셀/클릭 리다이렉트, provider webhook, batch delivery 성공/실패/skip/재시도/열람/클릭/반송/신고/수신 제외 상태 확인
- 가격 알림 구독 생성
- 목표가 도달 평가와 발송 큐 이벤트 저장
- 알림 발송 채널 설정, 큐 dispatch, 발송 시도/재시도 상태 기록
- 추천 만족도 피드백과 구매 의향 저장
- 베타 신청 리드 저장과 마스킹된 연락처 표시
- `X-SpecPilot-Key` 기반 워크스페이스 격리
- 출처 신뢰 등급, 캐시 TTL, 검수 필요 여부 표시
- 제휴 고지와 추천 공정성 정책 API
- Agent trace 확인
- LangGraph 단계별 trace span 저장과 운영 콘솔 조회
- Observability export outbox에 trace span과 품질 감사 payload 적재 및 dispatch/retry 상태 확인
- 운영 지표 확인
- 품질 점수, 예상 비용, 공개 차단 사유 확인
- 품질 회귀 모니터에서 최근/이전 품질, 비용 변화, provider 차단율 확인
- 베타 출시 준비도 점수와 다음 실행 항목 확인
- 시나리오별 베타 cohort와 자동 개선 백로그 확인
- cohort JSON/Markdown 리포트 export 확인
- 개선 백로그 담당자, 상태, SLA, 완료 요약 저장과 워크스페이스 격리 확인
- 소스 어댑터 상태 진단
- 실제 상품 URL/HTML 스냅샷 인입과 검수 큐 등록
- URL 모니터 등록, 수집 refresh 실행, refresh 성공/실패 이력 확인
- Due schedule preview와 due 모니터 refresh 실행
- Source provider 정책, robots/약관 승인, live fetch rate limit 게이트 확인
- 관리자 검수 콘솔
- Docker Compose 실행
- GitHub Actions 품질 게이트
- Neo4j 그래프 스키마 미리보기

## 운영 전 반드시 붙일 외부 연동

- 가격 비교/오픈마켓/공식 스토어의 공식 provider 계약과 외부 cron/Cloud Scheduler 배포
- 오픈마켓/공식 스토어 공식 API credential과 계약별 사용량 한도
- 벤치마크 출처 크롤링 또는 수동 검수 데이터
- 리뷰 수집/요약 파이프라인
- 가격 캐시 만료 정책
- 실제 이메일/SMS/웹훅 provider credential과 rate limit 정책
- LangSmith/OpenTelemetry 외부 export와 보존 기간 정책
- 실제 제휴 링크 연결과 비제휴 대안 노출 정책 검증

## 품질 게이트

| 게이트 | 기준 |
|---|---|
| 조건 진단 | 분석 전 예산, 목적, 필수 조건, 카테고리별 맥락 누락을 질문과 추천 조건으로 보강해야 한다 |
| 가격 정확도 | 모든 가격에 수집 시각, 출처, 실구매가 계산식이 있어야 한다 |
| 호환성 | 차단 이슈와 경고 이슈를 구분해야 한다 |
| 리뷰 표현 | 확정 표현 대신 반복 불만, 리스크 신호, 근거 수를 표시해야 한다 |
| 후보 근거 | 후보별 가격 계산, 리뷰 근거 수, 벤치마크, 호환성, 출처 신뢰를 한 번에 확인할 수 있어야 한다 |
| 옵션 검수 | 판매 페이지 옵션명, 장바구니 구성표, 최종 결제 화면의 핵심 사양을 대조할 수 있어야 한다 |
| 공유 브리프 | 공개 리포트 상단에서 판정, 핵심 근거, 리스크, 검토 질문, 복사용 문구를 바로 볼 수 있어야 한다 |
| 추천 공정성 | 가격, 목적, 호환성, 리뷰 점수를 분리해서 보여야 한다 |
| 구매 판정 | 구매 진행 가능, 가격 대기, 검수 후 구매가 가격/호환성/출처 신호에 따라 구분되어야 한다 |
| 대안 시나리오 | 최종 1순위 외에도 예산, 성능, 안전 우선 대안을 비교할 수 있어야 한다 |
| 조건 충족 | 필수 조건과 제외 조건을 후보별 충족, 확인 필요, 차단 상태로 보여야 한다 |
| 스트레스 테스트 | 예산 축소/확대와 조건 강화 시 선택 유지, 변경, 대기 판단을 보여야 한다 |
| 구매 타이밍 | 후보별 현재가, 목표가, 적정가 밴드, 재고/쿠폰 변동 리스크, 결제 트리거를 보여야 한다 |
| 구매 실행 | 결제 전 실행 단계, 판매자 확인 질문, 공유 검토 문구가 있어야 한다 |
| 출처 신뢰 | 출처별 신뢰도, 캐시 TTL, 검수 필요 여부가 응답과 UI에 노출되어야 한다 |
| 외부 URL 인입 | 내부망/private IP URL을 차단하고 상품 URL/HTML 스냅샷에서 추출한 가격 근거를 검수 큐에 등록해야 한다 |
| URL 모니터 | 반복 수집 대상 URL, refresh 성공/실패, live fetch 여부, 연결된 검수 항목을 운영 콘솔에서 확인할 수 있어야 한다 |
| Due refresh | cadence와 마지막 실행 시각 기준으로 지금 실행할 URL 모니터만 선별하고 실행할 수 있어야 한다 |
| Provider 정책 | live fetch는 승인된 host, robots/약관 승인, 시간당 rate limit을 통과해야 실행되어야 한다 |
| 제휴 고지 | 제휴 링크 여부와 추천 순위 계산 기준이 분리되어야 한다 |
| 공유 리포트 | 워크스페이스 소유자만 공유 토큰을 만들고 공개 페이지는 단일 리포트만 노출해야 한다 |
| 완료 리포트 | 저장 리포트를 템플릿과 수신자 그룹 기준으로 미리보기 후 batch 발송하고 provider 삽입용 추적 픽셀/클릭 리다이렉트, webhook secret 기반 provider 이벤트, unsubscribe 제외, 개별 delivery 성공/실패/skip/재시도/열람/클릭/반송/신고/수신 제외 상태가 워크스페이스별로 남아야 한다 |
| 알림 운영 | 목표가 도달 평가 결과와 발송 큐 이벤트가 저장되고 워크스페이스별로 격리되어야 한다 |
| 알림 발송 | 이메일/웹훅/SMS 채널 설정, 큐 dispatch, 발송 시도, 실패 재시도 기준이 워크스페이스별로 남아야 한다 |
| 베타 전환 | 추천 만족도, 구매 의향, 베타 리드가 저장되고 원문 연락처는 남기지 않아야 한다 |
| 베타 준비도 | 분석 실행, 공유 리포트 조회, 알림 연결, 피드백, 리드, 품질 차단 사유를 함께 본 launch readiness 판단이 있어야 한다 |
| 베타 cohort | 구매 시나리오별 리드, 피드백, 만족도, 구매 의향을 집계하고 개선 백로그를 자동 생성해야 한다 |
| 베타 운영 액션 | cohort 리포트 export와 백로그 상태/담당자/SLA/완료 요약을 워크스페이스별로 관리해야 한다 |
| 품질 감사 | 분석별 품질 점수, 예상 비용, 경고 수, 공개 차단 사유가 운영 콘솔에 표시되어야 한다 |
| 품질 회귀 | 최근/이전 분석 구간, 비용 변화, provider 차단율을 비교해 release health를 판단해야 한다 |
| 추적성 | 각 분석은 trace id와 단계별 span을 별도 저장하고 운영 API에서 워크스페이스별로 조회되어야 한다 |
| Observability | trace span과 품질 감사 payload가 외부 APM 연동 전 outbox에 저장되고 dispatch/retry 상태가 워크스페이스별로 격리되어야 한다 |
| 데이터 격리 | 다른 API 키의 저장 리포트와 trace는 조회되지 않아야 한다 |
| UI | 모바일에서 가로 스크롤 없이 분석과 비교표를 확인할 수 있어야 한다 |
| 배포 | Docker 이미지 빌드와 `/ready` 헬스체크가 통과해야 한다 |
| CI | Ruff, Pytest, Docker build가 main push에서 통과해야 한다 |

## 베타 테스트 시나리오

1. 영상 편집 + QHD 게이밍 데스크톱 200만원
2. 개발/Docker/AI 실험용 데스크톱 250만원
3. 휴대성 우선 영상 편집 노트북 200만원
4. 소규모 사업자 사무용 노트북 5대 구매 비교
5. 이미 가진 모니터/케이스/파워를 재사용하는 업그레이드 견적

## 출시 메시지

SpecPilot AI는 최저가 검색보다 구매 실패 가능성을 줄이는 데 집중한다. 사용자는 한 번의 요청으로 후보 5개, 최종 추천 3개, 제외 후보 2개, 가격 알림, 구매 전 체크리스트를 받을 수 있고 공유 가능한 구매 리포트로 다른 사람에게 검토를 요청할 수 있다.

공개 베타에서는 추천 결과가 실제 구매 의사결정에 도움이 되었는지 피드백과 구매 의향으로 확인하고, 베타 신청 리드를 통해 반복 구매 사용자군을 좁힌다.
