# SpecPilot AI 제품 기획서

## 1. 서비스 개요

SpecPilot AI는 컴퓨터 견적과 노트북 구매를 위한 AI 의사결정 에이전트다. 사용자가 예산, 사용 목적, 모니터 해상도, 선호 조건, 제외 조건을 입력하면 에이전트가 후보 견적과 노트북을 수집하고, 부품 호환성, 성능 병목, 가격, 리뷰, 벤치마크 근거를 함께 분석해 최종 추천 리포트를 생성한다.

핵심 컨셉은 최저가 검색이 아니라 구매 실패 가능성을 줄이는 것이다. 컴퓨터 구매에서 사용자가 실제로 어려워하는 지점은 CPU와 GPU의 균형, 메인보드 소켓, 파워 용량, 케이스 간섭, RAM/SSD 구성, 노트북 발열과 팬 소음, 옵션별 가격 차이다. SpecPilot AI는 이 판단을 구조화한다.

초기 제품명: SpecPilot AI

대상 MVP:

- 데스크톱 PC 견적
- 노트북 구매 비교

초기 대표 요청:

> 영상 편집과 게임용 데스크톱 200만원 안에서 맞춰줘. QHD 144Hz 모니터를 쓸 거고 업그레이드 여지도 있었으면 좋겠어.

## 2. 문제 정의

- 가격 비교 페이지는 부품별 최저가를 보여주지만 전체 견적의 호환성과 병목을 충분히 설명하지 못한다.
- 초보자는 CPU, GPU, 메인보드, RAM, 파워, 케이스의 조합이 맞는지 판단하기 어렵다.
- 노트북은 같은 모델이라도 RAM, SSD, GPU, 디스플레이 옵션에 따라 가격과 체감 성능이 크게 다르다.
- 리뷰와 커뮤니티 의견은 많지만 목적별로 어떤 불만이 중요한지 정리하기 어렵다.
- 할인, 배송비, 조립비, OS 포함 여부, 카드 혜택 때문에 실구매가 계산이 복잡하다.

## 5. LangGraph 에이전트 설계

LangGraph는 각 단계를 노드로 분리하고 상태 객체에 사용자 조건, 후보 제품, 가격 스냅샷, 리뷰 인사이트, 호환성 검증, 점수표, 출처, 최종 리포트를 누적한다.

노드 흐름:

1. Intent Parser - 자연어 요청을 예산, 목적, 카테고리, 필수 조건으로 구조화
2. Clarifier - 모호한 예산, 해상도, 휴대성, 조립 여부를 짧게 확인
3. Query Planner - 가격, 스펙, 벤치마크, 리뷰 검색 계획 생성
4. Product Collector - 데스크톱 견적 또는 노트북 후보 수집
5. Deduplicator - 모델명과 옵션 차이를 정규화하고 중복 제거
6. Compatibility Checker - CPU/보드/RAM/PSU/케이스 호환성 확인
7. Price Tracker - 배송비, 조립비, 쿠폰을 반영한 실구매가 계산
8. Review Analyzer - 반복 불만, 발열, 소음, AS, 리뷰 리스크 요약
9. Scoring Engine - 목적별 가중치로 종합 점수 산출
10. Verifier - 출처 누락, 가격 시점, 호환성 근거 부족 확인
11. Report Writer - 비교표와 최종 추천 생성

상태 모델:

```text
PurchaseState = {
  user_query,
  criteria,
  search_plan,
  candidates,
  normalized_products,
  compatibility_notes,
  price_snapshots,
  review_insights,
  scorecards,
  citations,
  verification_flags,
  final_report
}
```

## 6. 지식 그래프 설계

Neo4j는 단순 저장소가 아니라 구매 판단 근거를 관계로 연결하는 그래프 메모리로 사용한다.

```cypher
(:Build)-[:USES]->(:Component)
(:Build)-[:SOLD_AS]->(:Offer)-[:OFFERED_BY]->(:Seller)
(:Build)-[:CHECKED_BY]->(:CompatibilitySignal)
(:Component)-[:HAS_BENCHMARK]->(:Benchmark)
(:Laptop)-[:HAS_REVIEW]->(:Review)
```

대표 노드:

- Build: 정규화된 데스크톱 견적 조합
- Laptop: 노트북 완제품
- Component: CPU, GPU, RAM, SSD, PSU, Case
- Offer: 판매 가격, 배송비, 조립비, 쿠폰
- Seller: 가격 비교 출처, 오픈마켓, 공식 스토어
- Benchmark: 성능 근거
- Review: 사용자 리뷰와 전문 리뷰
- CompatibilitySignal: 소켓, 파워, 케이스 간섭 같은 검증 신호

## 12. 공개 베타 신뢰 정책

공개 베타에서는 추천 결과의 설득력보다 신뢰 장치를 먼저 노출한다. 사용자는 각 추천이 어떤 출처를 근거로 삼았는지, 가격이 얼마나 오래 캐시되는지, 어떤 근거가 관리자 검수 대상인지 확인할 수 있어야 한다.

현재 제품 구현:

- `/intake/diagnose`는 분석 전 구매 조건을 진단해 준비도 점수, 누락 슬롯, 추가 질문, 추천 필수/제외 조건, 정규화된 분석 요청을 반환한다.
- `/public/start-concierge`는 `/intake/diagnose`와 공개 온보딩 플레이북을 결합해 매칭 플레이북, 시작 마일스톤, 빠른 CTA, 전환 프롬프트를 반환한다.
- 루트 웹 UI는 조건 진단 버튼으로 추천 조건을 적용하거나 진단된 조건으로 바로 분석을 실행할 수 있다.
- Next.js `/launch` 공개 페이지는 첫 구매 진단 콘시어지 섹션에서 자연어 구매 문장, 예산, 목적, 필수/제외 조건으로 콘시어지를 실행하고 맞춤 플레이북, 시작 마일스톤, 빠른 CTA를 분석 폼 진입 전에 노출한다.
- 분리된 Next.js 웹사이트는 `/api/specpilot/intake-diagnose` 서버 프록시로 `/intake/diagnose`를 호출하고, 준비도 카드와 정규화 조건 적용 버튼을 분석 폼 앞에 노출한다.
- `/policy/trust` API로 캐시 정책, 제휴 고지, 공정성 규칙, 리뷰 표현 규칙을 공개한다.
- `/policy/privacy` API로 저장 데이터 범위, 연락처 마스킹, 공개 리포트 노출 범위, 보존 기간, 사용자 제어, 저장 금지 데이터를 공개한다.
- `/ops/data-governance`는 워크스페이스별 테이블 인벤토리, 원문 연락처 표면, 마스킹 표면, 보존 기간 초과 액션을 반환한다.
- `/beta/launch-gate`는 데이터 거버넌스 상태를 별도 게이트로 포함하고 원문 연락처 표면이나 보존 기간 초과 항목을 공개 확대 전 필수 액션으로 올린다.
- `report.source_trust`는 출처별 신뢰 등급, 신뢰도, freshness, cache TTL, 검수 필요 여부를 반환한다.
- `report.trust_policy`는 가격 만료 처리, 제휴 링크 고지, 비제휴 대안 노출 원칙을 반환한다.
- 웹 결과 화면은 출처 신뢰도와 캐시/제휴 정책을 분석 결과 옆에 표시한다.
- 신뢰도 0.8 미만 또는 리스크 플래그가 있는 수집 근거는 관리자 검수 큐로 들어간다.
- `/sources/ingest-url`은 외부 상품 URL 또는 HTML 스냅샷에서 제목, 표시 가격, 배송비, 쿠폰/카드 할인, 추정 실구매가, 재고 상태, 모델명 일치도, 본문 근거를 추출해 `SourceCandidate`와 검수 항목을 만든다.
- URL 인입은 내부망, localhost, private IP, 사용자 정보가 포함된 URL을 차단하고 실제 추천 반영 전 운영자 검수를 요구한다.
- 관리자 콘솔은 상품 URL과 기대 모델명을 입력해 실제 페이지 근거를 검수 큐로 보낼 수 있다.
- `/sources/monitors`는 반복 확인이 필요한 상품 URL을 워크스페이스별 모니터로 등록한다.
- `/sources/refresh`는 등록된 URL 모니터를 다시 수집하고 성공/실패, live fetch 여부, 연결된 검수 항목을 `source_refresh_runs` 이력으로 남긴다.
- `/sources/schedule`은 `cadence_minutes`와 마지막 실행 시각을 기준으로 due 모니터와 upcoming 모니터를 분리한다.
- `/sources/refresh-due`는 지금 실행해야 할 due 모니터만 수집해 외부 cron 또는 Cloud Scheduler가 호출할 운영 경계를 제공한다.
- 관리자 콘솔은 URL 모니터와 수집 refresh 이력을 보여줘 가격 근거 운영 상태를 추적하게 한다.
- Next.js 웹사이트는 URL 모니터 운영 섹션에서 `/sources/monitors`, `/sources/schedule`, `/sources/refresh-due`, `/sources/refresh-runs`, `/admin/reviews`를 묶어 due refresh, 최근 수집 이력, 검수 큐 승인/반려를 처리한다.
- `/sources/providers`는 live fetch 허용 host, robots/약관 검토 상태, credential 상태, 시간당 rate limit을 워크스페이스별 정책으로 저장한다.
- `/sources/providers/check`는 URL이 provider 정책과 rate limit을 통과하는지 확인하고, 정책이 없는 live fetch는 차단한다.
- live fetch 실행 시 provider 정책과 시간당 사용량 로그를 남겨 무분별한 외부 호출을 막는다.
- `/ops/integrations`는 가격 API, 오픈마켓, 공식 스토어, 리뷰 feed, 벤치마크, 이메일, webhook, observability, 제휴, scheduler provider의 credential 상태, rate limit, 보존 기간, 운영 증거를 저장한다.
- `/ops/integration-readiness`는 공개 전 필수 외부 연동을 mock/configured/verified/blocker 상태로 집계하고 미등록 핵심 연동을 출시 게이트 blocker로 올린다.

- `/beta/cohorts`는 구매 시나리오별 베타 테스트군을 만들고 persona, 키워드, 카테고리 기준으로 리드와 피드백을 집계한다.
- `/beta/backlog`는 readiness 경고, 낮은 만족도 피드백, 품질 차단 사유, 제품별 학습 인사이트에서 개선 백로그를 자동 생성한다.
- `/beta/backlog/{backlog_id}`는 자동 백로그에 운영 상태, 담당자, SLA, 완료 요약, 메모를 워크스페이스별로 덧씌운다.
- `/beta/backlog/summary`는 open, 진행, 완료, 지연, 마감임박, blocker 수와 최근 완료 요약을 반환한다.
- `/beta/cohorts/{cohort_id}/report`와 `/beta/cohorts/{cohort_id}/report.md`는 cohort 지표, 다음 액션, 열린 백로그를 JSON/Markdown으로 export한다.
- Next.js 웹사이트는 베타 운영 액션 섹션에서 cohort 생성, cohort 리포트 조회, 자동 개선 백로그 담당자/SLA/상태 갱신을 처리해 출시 게이트 전에 운영 액션을 닫게 한다.
- 운영 지표는 피드백 수, 평균 만족도, 구매 의향률, 실제 구매 결과 수, 구매 전환율, 최종가 평균 차이, 전환 금액, 제품별 학습 인사이트, 베타 리드 수, 공유 리포트 수, 공개 조회수를 함께 노출한다.
- 관리자 콘솔은 최근 사용자 피드백과 베타 리드를 별도 섹션으로 보여주고, 베타 출시 준비도, 출시 게이트, cohort 리포트 export, 개선 백로그 운영 상태와 다음 실행 항목을 노출한다.

| 베타 운영 액션 | cohort 리포트는 JSON/Markdown으로 export하고 백로그 상태, 담당자, SLA, 완료 요약, 메모는 워크스페이스별로 격리한다 |

## 최종 산출물 정의

사용자는 한 번의 요청으로 분석 전 조건 진단을 받고, 누락된 예산, 목적, 필수 조건, 데스크톱/노트북 맥락을 보강한 뒤 후보 5개를 수집하고, 최종 추천 3개와 제외 후보 2개를 비교표로 확인한다. 각 추천에는 실구매가, 핵심 장점, 반복 단점, 호환성 점검, 적합 사용자, 구매 전 확인할 점, 출처 링크가 포함된다. 운영자는 실제 상품 URL 또는 HTML 스냅샷을 인입해 가격과 페이지 근거를 검수 큐에 올리고, 반복 확인이 필요한 URL은 모니터로 등록해 cadence 기준 due refresh와 검수 큐 연결 상태를 추적한다. live fetch는 provider host 승인, robots/약관 검토, 시간당 rate limit을 통과한 경우에만 실행한다. 승인된 근거만 추천 신뢰 장치에 반영한다. 공유용 검토 브리프는 판정, 핵심 근거, 확인할 점, 검토 질문, 복사용 문구를 제공해 공개 리포트를 받은 사람이 긴 리포트 전에 판단하게 한다. 최종 리포트는 후보별 근거 팩으로 가격 계산, 리뷰 근거 수, 벤치마크, 호환성, 출처 신뢰를 한 화면에서 검증하게 한다. 옵션/사양 검수표는 판매 페이지 옵션명, 장바구니 구성표, 최종 결제 화면의 핵심 사양을 같은 기대값으로 대조하게 한다. 추가로 구매 진행 가능, 가격 대기, 검수 후 구매 중 하나의 판정과 확신도, 위험 플래그, 다음 행동을 제공한다. 구매 타이밍 윈도우는 후보별 현재가, 목표가, 적정가 밴드, 재고/쿠폰 변동 리스크, 결제 트리거를 묶어 즉시 결제와 가격 대기 판단을 분리한다. 저장 리포트 기반 구매 상담은 가격, 호환성, 리스크, 후보 비교, 결제 전 확인 질문에 대해 리포트 근거, 관련 후보, 상태, 확신도, 다음 행동을 붙여 답변하고 상담 이력을 워크스페이스별로 남긴다. 저장 리포트 기반 구매 링크는 후보별 제휴/비제휴 판매처, 가격, 쿠폰, 제휴 고지, 정책 경고를 공개 리포트에 노출하고 `/buy/{link_id}` redirect로 클릭 지표를 남긴다. 제휴 링크가 있으면 같은 후보의 비제휴 대안을 함께 제공해야 하며 추천 순위 계산에는 제휴 여부나 클릭 수를 직접 반영하지 않는다. 저장 리포트 기반 결제 전 검수는 최종 결제 금액, 옵션/사양 검수, 판매자 답변, 리스크 승인 누락을 기록해 결제 가능, 확인 필요, 보류 상태를 남긴다. 구매 결과 추적은 실제 구매, 이탈, 지연, 반품/취소 상태와 최종 결제 금액 차이, 만족도, 주문번호 마스킹값, 학습 신호를 저장해 추천 성과를 닫힌 루프로 검증한다. 학습 인사이트는 구매 결과, 결제 전 검수 차단, 피드백 만족도를 제품별 전환율, 반품률, 가격 신선도, 다음 개선 액션으로 묶어 운영자가 추천 노출과 근거 보강 우선순위를 정하게 한다. 저장된 완료 리포트는 발송 전 템플릿/수신자 그룹 기준으로 렌더링 미리보기를 확인한 뒤 운영 채널 outbox에 전달되고 provider 삽입용 추적 픽셀/클릭 리다이렉트 path를 발급한다. unsubscribe 대상은 skip delivery로 남으며 성공, 실패, 재시도 예정, 열람, 클릭, 반송, 신고, 수신 제외 상태가 기록된다. 목표가 도달 알림은 큐 이벤트와 채널별 dispatch 시도를 남겨 운영자가 발송 성공, 실패, 재시도 기준을 추적하게 한다. 또한 예산 절감, 성능 우선, 안전 우선 시나리오와 조건 충족 매트릭스, 예산/조건 스트레스 테스트, 구매 실행 패키지를 함께 제시해 사용자가 자기 우선순위와 입력 조건에 맞는 선택을 실제 구매 행동으로 전환하게 한다.

공개 베타 리포트와 운영 콘솔에는 추가로 출처 신뢰 등급, 캐시 TTL, 제휴 고지, 구매 링크 클릭 지표, 제휴/비제휴 대안 정책 경고, 검수 필요 여부, URL 모니터 due schedule, refresh 성공/실패 이력, 검수 큐 승인/반려 상태, 구매 상담 질문/답변/근거/다음 행동, 품질 감사 점수, 예상 비용, 품질 회귀 상태, provider 차단율, trace span, observability export outbox dispatch/retry, 외부 연동 provider의 mock/configured/verified/blocker 준비도, 완료 리포트 템플릿/수신자 그룹/batch delivery/engagement/provider webhook 상태, 공유 링크 조회수, 추천 만족도, 구매 의향, 실제 구매 결과, 구매 전환율, 최종가 차이, 제품별 학습 인사이트, 베타 신청 전환 지표, Free/Premium/Team 구독 의향, 예상 MRR, 연환산 매출, launch readiness 점수, readiness/품질 회귀/학습 인사이트/백로그 SLA/전환/발송/외부 연동 운영 상태를 묶은 출시 게이트 go/no-go, 시나리오별 cohort 집계, JSON/Markdown cohort 리포트 export, readiness/피드백/품질/학습 인사이트 기반 자동 개선 백로그, 백로그 담당자/상태/SLA/완료 요약/메모가 포함된다. 이는 최저가 추천보다 구매 실패 가능성을 줄이는 제품 포지션을 강화한다.
