# SpecPilot AI 제품 기획서

## 1. 서비스 개요

SpecPilot AI는 컴퓨터 견적과 노트북 구매를 위한 AI 의사결정 에이전트다. 사용자가 예산, 사용 목적, 모니터 해상도, 선호 조건, 제외 조건을 입력하면 에이전트가 후보 견적과 노트북을 수집하고, 부품 호환성, 성능 병목, 가격, 리뷰, 벤치마크 근거를 함께 분석해 최종 추천 리포트를 생성한다.

핵심 컨셉은 최저가 검색이 아니라 구매 실패 가능성을 줄이는 것이다. 컴퓨터 구매에서 사용자가 실제로 어려워하는 지점은 CPU와 GPU의 균형, 메인보드 소켓, 파워 용량, 케이스 간섭, RAM/SSD 구성, 노트북 발열과 팬 소음, 옵션별 가격 차이다. SpecPilot AI는 이 판단을 구조화한다.

초기 제품명: SpecPilot AI

대상 MVP:

- 데스크톱 PC 견적
- 노트북 구매 비교

초기 대표 요청:

> 영상 편집과 게임용 데스크톱 200만원 안에서 맞춰줘. QHD 144Hz 모니터를 쓸 거고 업그레이드 여지도 있었으면 좋겠어.

## 2. 문제 정의

- 가격 비교 페이지는 부품별 최저가를 보여주지만 전체 견적의 호환성과 병목을 충분히 설명하지 못한다.
- 초보자는 CPU, GPU, 메인보드, RAM, 파워, 케이스의 조합이 맞는지 판단하기 어렵다.
- 노트북은 같은 모델이라도 RAM, SSD, GPU, 디스플레이 옵션에 따라 가격과 체감 성능이 크게 다르다.
- 리뷰와 커뮤니티 의견은 많지만 목적별로 어떤 불만이 중요한지 정리하기 어렵다.
- 할인, 배송비, 조립비, OS 포함 여부, 카드 혜택 때문에 실구매가 계산이 복잡하다.

## 3. 사용자와 주요 시나리오

| 사용자 | 상황 | 성공 기준 |
|---|---|---|

| 크리에이터 | 영상 편집용 PC 또는 노트북을 고른다 | CPU/GPU/RAM/SSD 병목과 업그레이드 가치를 이해한다 |

## 5. LangGraph 에이전트 설계

LangGraph는 각 단계를 노드로 분리하고 상태 객체에 사용자 조건, 후보 제품, 가격 스냅샷, 리뷰 인사이트, 호환성 검증, 점수표, 출처, 최종 리포트를 누적한다.

노드 흐름:

1. Intent Parser - 자연어 요청을 예산, 목적, 카테고리, 필수 조건으로 구조화
2. Clarifier - 모호한 예산, 해상도, 휴대성, 조립 여부를 짧게 확인
3. Query Planner - 가격, 스펙, 벤치마크, 리뷰 검색 계획 생성
4. Product Collector - 데스크톱 견적 또는 노트북 후보 수집
5. Deduplicator - 모델명과 옵션 차이를 정규화하고 중복 제거
6. Compatibility Checker - CPU/보드/RAM/PSU/케이스 호환성 확인
7. Price Tracker - 배송비, 조립비, 쿠폰을 반영한 실구매가 계산
8. Review Analyzer - 반복 불만, 발열, 소음, AS, 리뷰 리스크 요약
9. Scoring Engine - 목적별 가중치로 종합 점수 산출
10. Verifier - 출처 누락, 가격 시점, 호환성 근거 부족 확인
11. Report Writer - 비교표와 최종 추천 생성

상태 모델:

```text
PurchaseState = {
  user_query,
  criteria,
  search_plan,
  candidates,
  normalized_products,
  compatibility_notes,
  price_snapshots,
  review_insights,
  scorecards,
  citations,
  verification_flags,
  final_report
}
```

## 6. 지식 그래프 설계

Neo4j는 단순 저장소가 아니라 구매 판단 근거를 관계로 연결하는 그래프 메모리로 사용한다.

```cypher
(:Build)-[:USES]->(:Component)
(:Build)-[:SOLD_AS]->(:Offer)-[:OFFERED_BY]->(:Seller)
(:Build)-[:CHECKED_BY]->(:CompatibilitySignal)
(:Component)-[:HAS_BENCHMARK]->(:Benchmark)
(:Laptop)-[:HAS_REVIEW]->(:Review)
```

대표 노드:

- Build: 정규화된 데스크톱 견적 조합
- Laptop: 노트북 완제품
- Component: CPU, GPU, RAM, SSD, PSU, Case
- Offer: 판매 가격, 배송비, 조립비, 쿠폰
- Seller: 가격 비교 출처, 오픈마켓, 공식 스토어
- Benchmark: 성능 근거
- Review: 사용자 리뷰와 전문 리뷰
- CompatibilitySignal: 소켓, 파워, 케이스 간섭 같은 검증 신호

| 평가 축 | 가중치 | 설명 |
|---|---:|---|

## 8. 화면 구성

| 화면 | 구성 요소 | 목적 |
|---|---|---|
| 요청 입력 | 카테고리, 예산, 용도, 해상도, 필수 조건, 제외 조건 | 좋은 검색 조건을 빠르게 만든다 |
| 조건 진단 | 분석 준비도 점수, 누락 슬롯, 추가 질문, 추천 조건 적용 버튼 | 애매한 요청을 좋은 분석 입력으로 바꾼다 |
| 분석 진행 | 수집 중인 출처, 후보 수, 호환성 체크, 리뷰 수 | 에이전트가 무엇을 하는지 투명하게 보여준다 |
| 추천 결과 | TOP 3 카드, 비교표, 가격, 호환성 배지 | 최종 선택을 빠르게 돕는다 |
| 공유용 검토 브리프 | 판정, 핵심 근거, 확인할 점, 검토 질문, 복사용 문구 | 공유받은 사람이 바로 판단하게 한다 |
| 후보별 근거 팩 | 가격 계산, 리뷰 근거 수, 벤치마크, 호환성, 출처 신뢰 카드 | 추천 근거를 한 화면에서 검증하게 한다 |
| 옵션/사양 검수표 | CPU, GPU, RAM, SSD, 파워, 디스플레이 등 핵심 사양 기대값 | 옵션 불일치 구매를 막는다 |
| 구매 판정 | 판정 라벨, 확신도, 위험 플래그, 다음 행동 | 지금 결제할지 기다릴지 결정하게 한다 |
| 대안 시나리오 | 예산 절감, 성능 우선, 안전 우선 카드 | 다른 우선순위의 합리적 선택지를 보여준다 |
| 조건 충족 매트릭스 | 필수 조건과 제외 조건의 충족률, 경고, 차단 수 | 내 조건을 실제로 만족하는지 검증하게 한다 |
| 예산/조건 스트레스 테스트 | 예산 축소, 예산 확대, 조건 강화 카드 | 선택이 가격과 조건 변화에도 버티는지 확인하게 한다 |
| 구매 타이밍 윈도우 | 현재가, 목표가, 적정가 밴드, 변동 리스크, 결제 트리거 | 지금 결제할지 기다릴지 판단하게 한다 |
| 구매 실행 패키지 | 결제 전 실행 단계, 판매자 확인 질문, 공유 검토 문구 | 추천을 실제 구매 행동으로 전환하게 한다 |
| 상세 근거 | 부품 구성, 벤치마크, 리뷰 요약, 반복 불만 | 추천 근거를 검증하게 한다 |
| 공개 리포트 | 공유 링크, TOP 3, 비교표, 검증 플래그, 출처 신뢰도 | 가족, 동료, 커뮤니티 검토로 확산되게 한다 |
| 가격 알림 | 목표가, 재조회 주기, 알림 채널 | 즉시 구매하지 않는 사용자를 잡는다 |
| 운영 감사 | 품질 점수, 예상 비용, 경고, 공개 차단 사유 | 공개 후 문제 분석과 개선 속도를 높인다 |
| 베타 피드백 | 만족도, 구매 의향, 개선 요청, 베타 신청 | 추천 결과가 실제 구매 판단으로 이어지는지 검증한다 |

| 호환성 오판 | 구매 실패 | Verifier 노드, 규칙 기반 검증, 사용자 재확인 |
| 리뷰 오판 | 분쟁 가능성 | 확정 표현 금지, 리스크 신호와 근거만 표시 |
| LLM 환각 | 잘못된 스펙 제시 | 구조화 데이터 우선, 출처 없는 주장 제거 |

## 12. 공개 베타 신뢰 정책

공개 베타에서는 추천 결과의 설득력보다 신뢰 장치를 먼저 노출한다. 사용자는 각 추천이 어떤 출처를 근거로 삼았는지, 가격이 얼마나 오래 캐시되는지, 어떤 근거가 관리자 검수 대상인지 확인할 수 있어야 한다.

현재 제품 구현:

- `/intake/diagnose`는 분석 전 구매 조건을 진단해 준비도 점수, 누락 슬롯, 추가 질문, 추천 필수/제외 조건, 정규화된 분석 요청을 반환한다.
- 루트 웹 UI는 조건 진단 버튼으로 추천 조건을 적용하거나 진단된 조건으로 바로 분석을 실행할 수 있다.
- `/policy/trust` API로 캐시 정책, 제휴 고지, 공정성 규칙, 리뷰 표현 규칙을 공개한다.
- `report.source_trust`는 출처별 신뢰 등급, 신뢰도, freshness, cache TTL, 검수 필요 여부를 반환한다.
- `report.trust_policy`는 가격 만료 처리, 제휴 링크 고지, 비제휴 대안 노출 원칙을 반환한다.
- 웹 결과 화면은 출처 신뢰도와 캐시/제휴 정책을 분석 결과 옆에 표시한다.
- 신뢰도 0.8 미만 또는 리스크 플래그가 있는 수집 근거는 관리자 검수 큐로 들어간다.
- `/alerts/evaluate`는 활성 알림 구독을 평가하고 목표가 도달 시 발송 큐 이벤트를 만든다.
- `/alerts/events`는 워크스페이스별 알림 발송 이벤트를 조회한다.
- 운영 지표는 알림 구독 수, 발송 이벤트 수, 트리거 수를 함께 노출한다.
- `quality_audit`는 분석별 품질 점수, 예상 소스 호출, 예상 토큰/비용, 경고 수, 공개 차단 사유를 포함한다.
- `/ops/quality`는 워크스페이스별 품질 감사 대시보드를 반환한다.
- 관리자 콘솔은 품질/비용 감사 섹션에서 최근 분석의 공개 가능성을 보여준다.
- `/reports/{report_id}/share`는 워크스페이스 소유자가 저장 리포트에 공개 토큰을 발급하게 한다.
- `/public/reports/{share_token}`과 `/r/{share_token}`은 API 키 없이 단일 공개 리포트만 조회하게 한다.
- 공유 해제 시 기존 공개 토큰은 404가 되며 워크스페이스 내부 리포트 목록은 노출되지 않는다.
- `report.purchase_decision`은 점수, 목표가, 예산 초과, 호환성 차단, 출처 검수 신호를 종합해 구매 진행 가능, 가격 대기, 검수 후 구매로 나눈다.
- 웹 결과 화면과 공개 리포트는 구매 판정, 확신도, 결제 전 체크리스트를 추천 결과 앞에 표시한다.
- `report.share_brief`는 공유받은 사람이 바로 검토할 판정, 핵심 근거, 리스크, 질문, 복사용 문구를 반환한다.
- 웹 결과 화면과 공개 리포트는 공유용 검토 브리프를 추천 카드 직후에 표시한다.
- `report.scenario_options`는 예산 절감, 성능 우선, 안전 우선 기준의 대안 후보와 트레이드오프를 반환한다.
- 웹 결과 화면과 공개 리포트는 TOP 3 외에도 대안 시나리오 카드를 표시한다.
- `report.criteria_matches`는 필수 조건과 제외 조건의 후보별 충족률, 경고, 차단 사유를 반환한다.
- 웹 결과 화면과 공개 리포트는 조건 충족 매트릭스를 비교표와 함께 표시한다.
- `report.stress_tests`는 예산 10% 절감, 예산 10% 여유, 조건 강화 시 선택 유지, 변경, 대기 판단을 반환한다.
- 웹 결과 화면과 공개 리포트는 스트레스 테스트 카드를 대안 시나리오 다음에 표시한다.
- `report.deal_windows`는 후보별 현재가, 목표가, 적정가 밴드, 변동 리스크, 대기 이유, 결제 트리거를 반환한다.
- 웹 결과 화면과 공개 리포트는 구매 타이밍 윈도우를 공유 브리프 다음에 표시한다.
- `report.evidence_packs`는 후보별 가격 계산, 리뷰 근거 수, 벤치마크, 호환성, 출처 신뢰, 검수 필요 여부를 반환한다.
- 웹 결과 화면과 공개 리포트는 근거 팩을 스트레스 테스트 다음에 표시한다.
- `report.option_audits`는 후보별 핵심 사양 기대값, 확인 방법, 옵션 불일치 리스크, 구매 차단 항목을 반환한다.
- 웹 결과 화면과 공개 리포트는 옵션/사양 검수표를 근거 팩 다음에 표시한다.
- `report.execution_plan`은 결제 전 실행 단계, 판매자 확인 질문, 공유 검토 문구를 반환한다.
- 웹 결과 화면과 공개 리포트는 구매 실행 패키지를 구매 판정 직후 표시한다.
- `/feedback`은 추천 만족도, 구매 의향, 선택 후보, 개선 요청을 저장한다.
- `/beta/leads`는 베타 신청 리드를 저장하되 이메일 원문은 저장하지 않고 마스킹된 값만 남긴다.
- 운영 지표는 피드백 수, 평균 만족도, 구매 의향률, 베타 리드 수를 함께 노출한다.
- 관리자 콘솔은 최근 사용자 피드백과 베타 리드를 별도 섹션으로 보여준다.

정책 기준:

| 항목 | 기준 |
|---|---|
| 조건 진단 | 예산, 목적, 필수 조건, 카테고리별 맥락이 부족하면 분석 전 질문과 보강 조건을 제시한다 |
| 가격 캐시 | 가격 소스는 45-180분 단위로 캐시하고 만료 시 구매 직전 재확인을 요구한다 |
| 리뷰 근거 | 단정 표현 대신 반복 불만, 리스크 신호, 근거 수를 표시한다 |
| 벤치마크 | 목적별 성능 판단에 쓰되 가격 순위보다 우선하지 않는다 |
| 후보 근거 팩 | 가격, 리뷰, 벤치마크, 호환성, 출처 신뢰를 후보별로 묶어 검증 가능하게 한다 |
| 제휴 고지 | 제휴 링크가 붙더라도 추천 순위는 목적, 가격, 호환성, 리뷰 신뢰도로 계산한다 |
| 관리자 검수 | 낮은 신뢰도와 리스크 플래그가 있는 근거는 공개 추천 전에 검수한다 |
| 공유 리포트 | 공개 토큰은 단일 저장 리포트만 열며 소유 워크스페이스만 생성과 해제가 가능하다 |
| 구매 판정 | 중대한 호환성 또는 출처 검수 신호가 있으면 결제 전 검수로 판정한다 |
| 대안 시나리오 | 1순위를 강요하지 않고 예산, 성능, 안전 우선 관점의 합리적 대안을 함께 제공한다 |
| 스트레스 테스트 | 예산과 조건 변화에 취약한 추천은 유지, 변경, 대기 판단을 명확히 노출한다 |
| 구매 타이밍 | 현재가, 목표가, 적정가 밴드, 재고/쿠폰 변동 리스크를 기준으로 즉시 결제와 가격 대기를 구분한다 |
| 알림 발송 | 실제 외부 채널 전송 전 발송 큐 이벤트를 저장하고 개인정보는 마스킹한다 |
| 품질 감사 | 품질 점수와 예상 비용을 저장하고 공개 차단 사유가 있으면 운영자가 먼저 확인한다 |
| 베타 피드백 | 연락처와 이메일 원문을 저장하지 않고 추천 만족도와 구매 의향을 전환 지표로 본다 |

## 최종 산출물 정의

사용자는 한 번의 요청으로 분석 전 조건 진단을 받고, 누락된 예산, 목적, 필수 조건, 데스크톱/노트북 맥락을 보강한 뒤 후보 5개를 수집하고, 최종 추천 3개와 제외 후보 2개를 비교표로 확인한다. 각 추천에는 실구매가, 핵심 장점, 반복 단점, 호환성 점검, 적합 사용자, 구매 전 확인할 점, 출처 링크가 포함된다. 공유용 검토 브리프는 판정, 핵심 근거, 확인할 점, 검토 질문, 복사용 문구를 제공해 공개 리포트를 받은 사람이 긴 리포트 전에 판단하게 한다. 최종 리포트는 후보별 근거 팩으로 가격 계산, 리뷰 근거 수, 벤치마크, 호환성, 출처 신뢰를 한 화면에서 검증하게 한다. 옵션/사양 검수표는 판매 페이지 옵션명, 장바구니 구성표, 최종 결제 화면의 핵심 사양을 같은 기대값으로 대조하게 한다. 추가로 구매 진행 가능, 가격 대기, 검수 후 구매 중 하나의 판정과 확신도, 위험 플래그, 다음 행동을 제공한다. 구매 타이밍 윈도우는 후보별 현재가, 목표가, 적정가 밴드, 재고/쿠폰 변동 리스크, 결제 트리거를 묶어 즉시 결제와 가격 대기 판단을 분리한다. 또한 예산 절감, 성능 우선, 안전 우선 시나리오와 조건 충족 매트릭스, 예산/조건 스트레스 테스트, 구매 실행 패키지를 함께 제시해 사용자가 자기 우선순위와 입력 조건에 맞는 선택을 실제 구매 행동으로 전환하게 한다.

공개 베타 리포트에는 추가로 출처 신뢰 등급, 캐시 TTL, 제휴 고지, 검수 필요 여부, 품질 감사 점수, 예상 비용, 공유 링크 조회수, 추천 만족도, 구매 의향, 베타 신청 전환 지표가 포함된다. 이는 최저가 추천보다 구매 실패 가능성을 줄이는 제품 포지션을 강화한다.
